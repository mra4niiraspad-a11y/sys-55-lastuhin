Задание 1. СУБД
Кейс
Крупная строительная компания, которая также занимается проектированием и девелопментом, решила создать правильную архитектуру для работы с данными. Ниже представлены задачи, которые необходимо решить для каждой предметной области.

Какие типы СУБД, на ваш взгляд, лучше всего подойдут для решения этих задач и почему?

1.1. Бюджетирование проектов с дальнейшим формированием финансовых аналитических отчётов и прогнозирования рисков. СУБД должна гарантировать целостность и чёткую структуру данных.

Ответ :  Реляционная СУБД (например, PostgreSQL, Oracle, MS SQL Server). Гарантирует целостность данных через транзакции (ACID), поддерживает чёткую структуру в виде таблиц с заданными схемами — идеально для финансовых расчётов и аналитики.

1.2. Под каждый девелоперский проект создаётся отдельный лендинг, и все данные по лидам стекаются в CRM к маркетологам и менеджерам по продажам. Какой тип СУБД лучше использовать для лендингов и для CRM? СУБД должны быть гибкими и быстрыми.

Ответ :  Для лендингов — NoSQL документо‑ориентированная СУБД (MongoDB, CouchDB): гибкость схемы позволяет быстро менять структуру данных под разные лендинги. Для CRM — реляционная СУБД (PostgreSQL, MySQL): обеспечивает целостность данных о клиентах и сделках, поддерживает сложные запросы для аналитики.

1.3. Отдел контроля качества решил создать базу по корпоративным нормам и правилам, обучающему материалу и так далее, сформированную согласно структуре компании. СУБД должна иметь простую и понятную структуру.

Ответ :  Иерархическая или реляционная СУБД. Иерархическая (например, XML‑базы) подойдёт для древовидной структуры компании. Реляционная (PostgreSQL) — если нужна простота и структурированность с таблицами «Отделы», «Правила», «Обучение».

1.4. Департамент логистики нуждается в решении задач по быстрому формированию маршрутов доставки материалов по объектам и распределению курьеров по маршрутам с доставкой документов. СУБД должна уметь быстро работать со связями.

Ответ :  Графовая СУБД (Neo4j, Amazon Neptune). Оптимизирована для работы со связями (маршруты, курьеры, объекты), позволяет быстро выполнять запросы на поиск оптимальных путей и распределение ресурсов.

Задание 2. Транзакции

2.1. Пользователь пополняет баланс счёта телефона, распишите пошагово, какие действия должны произойти для того, чтобы транзакция завершилась успешно. Ориентируйтесь на шесть действий.

Начало транзакции: система запускает транзакцию, блокируя счёт пользователя для предотвращения параллельных изменений.

Проверка баланса карты: система проверяет, достаточно ли средств на банковской карте пользователя.

Списание суммы: если средств достаточно, сумма списывается с карты и временно резервируется.

Обновление баланса телефона: система обновляет баланс телефонного счёта пользователя, добавляя зачисленную сумму.

Фиксация транзакции: система фиксирует изменения (COMMIT), делая их постоянными в базе данных.

Подтверждение операции: пользователю отправляется уведомление об успешном пополнении баланса (SMS, email или в интерфейсе приложения).

Если на любом шаге возникает ошибка (недостаточно средств, сбой связи), выполняется откат транзакции (ROLLBACK), и изменения отменяются.

Задание 3. SQL vs NoSQL
3.1. Напишите пять преимуществ SQL-систем по отношению к NoSQL.

Пять преимуществ SQL‑систем перед NoSQL:

Гарантированная целостность данных (ACID): транзакции обеспечивают атомарность, согласованность, изоляцию и надёжность — критично для финансовых и учётных систем.

Чёткая схема данных: жёсткая структура таблиц упрощает проектирование, поддержку и понимание базы данных, снижает риск ошибок.

Мощные запросы с JOIN: SQL позволяет легко объединять данные из нескольких таблиц, что удобно для сложной аналитики и отчётов.

Стандартизированный язык запросов: SQL — универсальный стандарт, знакомый большинству разработчиков; NoSQL‑системы часто используют проприетарные API или языки запросов.

Развитая экосистема инструментов: множество готовых решений для резервного копирования, мониторинга, миграции и администрирования, оптимизированных под реляционные СУБД.

Задание 4. Кластеры
Необходимо производить большое количество вычислений при работе с огромным количеством данных, под эту задачу выделено 1000 машин.

На основе какого критерия будете выбирать тип СУБД и какая модель распределённых вычислений здесь справится лучше всего и почему?

Критерий выбора СУБД: способность масштабироваться горизонтально (добавление узлов) и обрабатывать большие объёмы данных параллельно, с минимальной задержкой и высокой отказоустойчивостью.

Тип СУБД: распределённая NoSQL‑СУБД (например, Apache Cassandra, HBase) или колоночная аналитическая СУБД (ClickHouse, Google BigQuery). Они:

масштабируются на тысячи узлов;

обеспечивают высокую пропускную способность для пакетной и потоковой обработки;

устойчивы к отказам отдельных машин (данные реплицируются).

Модель распределённых вычислений: MapReduce (в экосистеме Hadoop) или Spark.

Почему:

MapReduce/Spark разбивают вычисления на мелкие задачи, распределяя их по 1 000 машин. Это позволяет параллельно обрабатывать огромные наборы данных.

Spark предпочтительнее для итеративных вычислений и машинного обучения: он кэширует данные в памяти, что ускоряет обработку по сравнению с MapReduce (который записывает промежуточные результаты на диск).

Сочетание NoSQL‑СУБД (для хранения) и Spark (для вычислений) создаёт отказоустойчивую, масштабируемую среду для анализа больших данных (Big Data).
