Задание 1. Анализ логов Suricata и комментарий результата.
Выполнено сканирование с использованием ACK‑сканирования (проверка фильтрации трафика):
sudo nmap -sA 10.0.2.15
Выполните TCP‑подключение (полное трёхэтапное рукопожатие):
sudo nmap -sT 10.0.2.15
Выполните SYN‑сканирование (полуоткрытое сканирование):
sudo nmap -sS 10.0.2.15
Выполните сканирование для определения версий служб:
sudo nmap -sV 10.0.2.15

Результат:
На скриншоте https://github.com/mra4niiraspad-a11y/sys-55-lastuhin/blob/d90d3117d2c04324cf78261808370876ae6d11f9/VirtualBox_ubu_23_06_2026_11_49_28.png в логах Suricata (/var/log/suricata/fast.log) фиксируются события с сигнатурой [1:1000001:1] Possible SYN Portscan — это детектирование SYN‑сканирования (попыток установить TCP‑соединение без завершения рукопожатия).

Ключевые детали из логов:

Тип события: сканирование портов по протоколу TCP (указано {TCP}).
Источник: IP‑адрес 10.0.2.15 (атакующая система).
Назначение: IP‑адрес 192.168.56.10, порт 80 (целевая система, вероятно, веб‑сервер).
Временные метки: события идут сериями с интервалом ~1 минута (например, 11:45:36, 11:46:53, 11:47:53), что типично для автоматизированного сканирования.
Классификация и приоритет: [Classification: (null)], [Priority: 3] — правило сработало, но без специальной классификации; приоритет средний.
Комментарий: события напрямую коррелируют с запуском nmap -sS 10.0.2.15 (SYN‑сканирование). Suricata корректно распознала шаблон сетевой активности как потенциальный порт‑скан. Наличие повторяющихся срабатываний подтверждает многократные попытки сканирования одного и того же сервиса (порт 80).

В логах Fail2Ban (/var/log/fail2ban.log) на момент снимка явных записей о блокировках не видно — вероятно, Fail2Ban либо ещё не зафиксировал достаточное количество неудачных попыток для срабатывания, либо не настроен на мониторинг соответствующего сервиса (например, веб‑сервера на порту 80).
